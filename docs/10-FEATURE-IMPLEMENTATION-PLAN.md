# 10-Feature Implementation Plan

**Generated:** 2026-03-08  
**Codebase snapshot:** Post Phase 1-5 (readiness score, contribution room, spending phases, what-if drawer, plan completeness all already implemented)

---

## Summary

This plan adds 10 features to the RetireePlan app across 6 implementation phases. Features are ordered by dependency depth: foundational engine/schema work first, then API endpoints, then UI, then cross-cutting features. Each step lists exact file paths so an orchestrator can detect conflicts and parallelise non-overlapping work.

Key dependencies between features:
- **Feature 5 (Real Estate)** requires a schema migration that Feature 4 (Asset Allocation) also touches → sequence schema work
- **Feature 1 (Insights Engine)** consumes outputs from Features 3, 4, and the existing readiness score → implement last of the engine features
- **Feature 7 (PDF Report)** consumes data from Features 1, 3, 8 → implement after those
- **Feature 8 (Goals)** requires a new Prisma model + engine + full-stack → independent track
- **Feature 10 (Market Data Refresh)** extends existing market-data module → independent track

### Step 1.1 — Real Estate: Prisma Schema Extension (Feature 5)

Add a new `RealEstate` model and link it to `Household`.

**Files to create:**

- [prisma/schema.prisma](prisma/schema.prisma) — Add `RealEstate` model:
  ```prisma
  model RealEstate {
    id                  String    @id @default(cuid())
    name                String                          // "Primary Residence", "Cottage", etc.
    propertyType        String                          // 'PRIMARY_RESIDENCE' | 'RENTAL' | 'VACATION'
    currentValue        Float

    /// Link to household

    updatedAt           DateTime  @updatedAt
  }
  ```
- [prisma/schema.prisma](prisma/schema.prisma) — Add `realEstateProperties RealEstate[]` to `Household` model

**Implementation steps:**
1. Add `RealEstate` model and relation to `prisma/schema.prisma`

### Step 1.6 — Insights Engine: Core Rule Engine (Feature 1)

**Files to create:**
- `packages/finance-engine/src/insights/insights-engine.ts` — Pure functions:
  ```typescript
  export interface InsightInput {
    // Member data
    currentAge: number;
    retirementAge: number;
    annualIncome: number;
    province: string;
    // Balances
    rrspBalance: number;
    tfsaBalance: number;
    nonRegBalance: number;
    // Contribution room
    unusedTfsaRoom: number;
    unusedRrspRoom: number;
    // Benefits
    cppStartAge: number;
    oasStartAge: number;
    // Projection outputs
    oasClawbackYears: number[];   // ages where OAS clawback > 0
    projectedIncomeAtRetirement: number;
    // Couple data (optional)
    hasSpouse: boolean;
    spouseAge?: number;
    spouseIncome?: number;
    // Engine outputs
    meltdownSavings?: number;     // from RRSP meltdown optimizer
    rrifConversionAge: number;
  }
  
  export interface Insight {
    id: string;                     // e.g. 'rrsp-meltdown', 'oasc-clawback'
    title: string;
    description: string;
    dollarImpact: number;           // estimated $ benefit
    priority: 'high' | 'medium' | 'low';
    linkTo: string;                 // route path, e.g. '/tax-analytics'
    category: 'tax' | 'benefits' | 'investment' | 'estate';
  }
  
  export function generateInsights(input: InsightInput): Insight[]
  // Returns max 5 insights sorted by dollarImpact descending
  ```
  
  Rule implementations:
  1. RRSP meltdown suggestion: RRSP > 2× annualIncome AND age 55–68
  2. OAS clawback: any year with oasClawback > 0 → shift withdrawals
  3. TFSA room: unusedTfsaRoom > 10,000 AND nonRegBalance > 0
  4. RRIF reminder: age ≥ rrifConversionAge - 3
  5. CPP timing: age ≥ 60 AND cppStartAge not yet reached
  6. Pension splitting: hasSpouse AND income delta > $20k

**Files to modify:**
- [packages/finance-engine/src/index.ts](packages/finance-engine/src/index.ts) — Add `export * from './insights/insights-engine.js';`

**Can run in parallel with:** All other Step 1.x items.

## Edge Cases to Handle

### Feature 1 — Insights Engine
- Handle single-member households (no pension splitting insight)
- Handle no projection data yet (skip OAS clawback detection, return fewer insights)
- Cap at 5 insights; if fewer rules trigger, return fewer
- Dollar impact = 0 for non-quantifiable insights (CPP timing suggestion) — use estimated value

### Feature 2 — Phased Spending
- Spending phases with overlapping `fromAge` — last one wins (engine already handles this)
- Factor = 0 should warn user (0% spending is likely a mistake)
- Validate phases are ordered by `fromAge`

### Feature 3 — Contribution Room
- Member with no `rrspContributionRoom` set → show prompt to enter CRA NOA data
- Age > 71 → RRSP room is 0 (convert to RRIF, no more contributions)
- TFSA for members born before 1991 — room starts at year they turned 18 or 2009, whichever is later (engine already handles this)

### Feature 4 — Asset Allocation
- Allocation percentages must sum to 100% (or be null for "unset")
- Accounts with no allocation set → use scenario's default return rate (backward compatible)
- Glide path must have strictly increasing ages

### Feature 5 — Real Estate
- Primary residence: capital gains exempt (principal residence exemption) — engine must not tax it
- Rental property net income: must deduct expenses before adding to income sources projection
- Downsizing: `sellAtAge` must be ≥ current age
- Multiple primary residences → warn (only one PRE allowed)

### Feature 6 — Drawdown Waterfall
- Projection data may have 0 or negative balances → clamp bars to 0
- Long projections (40+ years) → downsample to max ~40 bars for readability
- Animation performance: use D3 `transition` with `requestAnimationFrame`, not `setInterval`

### Feature 7 — PDF Report
- Missing data fields (no insights, no estate calc) → omit those sections gracefully
- PDF rendering with @react-pdf/renderer doesn't support D3 → use SVG primitives (Svg, Path, Rect) for charts (existing pattern in `NetWorthChart`, `IncomeExpenseChart`)
- Very long scenario names → truncate in table headers

### Feature 8 — Goals
- Goal with no `targetAge` → evaluate at life expectancy
- Monte Carlo with 0 goals → return empty results
- Goal amount = 0 → always 100% success
- Circular dependency: goal "retire at 60" affects projection which evaluates goals — resolve by running projection first, then evaluating goals against its output

### Feature 9 — Milestone Templates
- Template amounts are defaults — user must be able to edit before saving
- Templates should use current year dollars (inflation-adjusted later by engine)
- Duplicate detection: warn if milestone with same name already exists

### Feature 10 — Market Data Refresh
- Bank of Canada API may be down → graceful fallback with "last known" values
- Year boundary: "first login of year" detection via `localStorage.getItem('lastRefreshYear')` compared to `new Date().getFullYear()`
- Constants update: TFSA limit for future years may not be announced yet → use last known limit as default (engine already does this)

## Open Questions

1. **Feature 5 (Real Estate):** Should rental income be modeled as a new `IncomeSource` type automatically when a rental property is added, or kept separate? Creating an auto-linked income source is cleaner for projections but adds complexity.

2. **Feature 4 (Asset Allocation):** Should the glide path builder be per-scenario or per-account? Current engine supports per-scenario via `glidePathSteps`. Per-account would require engine changes.

3. **Feature 8 (Goals):** Should Monte Carlo trials be run fresh for each goal evaluation, or reuse the most recent household MC run? Fresh = accurate but slow; reuse = fast but potentially stale.

4. **Feature 1 (Insights):** Should insights be cached server-side (compute on save, store in DB) or computed on-demand each time the dashboard loads? On-demand is simpler but slower.

5. **Feature 7 (PDF):** Should the PDF include Monte Carlo fan chart? The existing `MonteCarloFanChart` PDF component already exists — confirm it should be in the executive summary or a dedicated sim page.

6. **Feature 10 (Market Data):** The Bank of Canada Valet API endpoints already exist (`POST /market-data/fetch-boc`). Should the annual refresh trigger this automatically, or require explicit user action?
